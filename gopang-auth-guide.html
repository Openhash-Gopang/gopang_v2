고팡 계정 하나로
모든 서비스를 이용한다

자신의 스마트폰을 지문이나 패턴 등으로 연 뒤, 폰에 설치된 수십 개의 앱을 제한없이 이용할 수 있습니다. 웹은 어떤가요? 국세청, 은행, 기업, 특허청 등 제각각의 로그인 절차와 별의별 인증서를 요구합니다. 고팡은 마치 스마트폰처럼 기존의 웹을 하나의 인증 방법으로 열고, 그 앱인 수백만 개 웹사이트를 AI로 재구성합니다. 말하자면, 현실 세계의 웹을 그대로 투사한 "AI 평행 세계의 웹"입니다.

고팡은 단순한 앱이 아닙니다. K-Police(치안), K-School(교육), K-Law(법률), K-Health(의료) 등 수십 개의 하위 서비스로 이루어진 거대한 생태계입니다. 이 서비스들은 각기 다른 주소(도메인)에 있지만, 사용자는 계정을 하나만 만들면 모든 서비스를 자유롭게 이용할 수 있습니다.

어떻게 가능할까요? 이것을 "인증 프로세스 공유"라고 합니다.

🏢 아파트 단지 마스터키에 비유하면:
고팡은 하나의 대형 아파트 단지입니다. 경비실(고팡)에서 한 번 신분 확인을 마치면, 단지 안의 헬스장(K-Health), 도서관(K-School), 경찰초소(K-Police) 어디든 같은 마스터키로 출입할 수 있습니다. 각 시설마다 따로 신분증을 제시할 필요가 없습니다.

이 방식을 SSO(Single Sign-On, 단일 로그인)라고 부릅니다. 전 세계 수많은 서비스가 이 방식을 사용합니다. 구글 계정 하나로 유튜브·지메일·구글맵을 모두 이용하는 것과 같은 원리입니다.

사용자 입장에서 본 전체 흐름

Step 1 — 고팡에서 딱 한 번 등록합니다

처음 고팡(gopang.net)에 접속하면 두 가지를 설정합니다. 하나는 기기 인식입니다. 고팡은 사용자의 스마트폰이나 컴퓨터를 조용히 분석해 고유한 "기기 지문"을 만듭니다. 다른 하나는 4단어 시드입니다. 기기를 잃어버렸을 때 복원하기 위한 비밀 열쇠입니다. 선택적으로 얼굴 등록도 할 수 있습니다.

🔑 지문과 비밀 암호에 비유하면:
기기 지문은 여러분의 손가락 지문과 같습니다. 같은 기기라면 언제나 동일한 지문이 나옵니다. 4단어 시드는 은행 금고의 비밀번호와 같습니다. 기기가 바뀌어도 이 암호를 알면 다시 본인임을 증명할 수 있습니다.

Step 2 — K-Police(또는 다른 하위 서비스)에 접속합니다

사용자가 police.gopang.net에 접속하면, K-Police는 "이 사람이 누구인지" 고팡에게 조용히 물어봅니다. 사용자는 이 과정을 전혀 알아채지 못합니다. 1~3초 안에 자동으로 완료되기 때문입니다.

📞 호텔 내선 전화에 비유하면:
호텔 레스토랑(K-Police)에 들어서면, 레스토랑 직원이 프런트(고팡)에 내선을 걸어 "507호 투숙객 맞나요?"라고 확인합니다. 손님(사용자)은 아무것도 할 필요가 없습니다. 프런트가 "맞습니다"라고 답하면 바로 자리를 안내받습니다.

Step 3 — 처음 접속이라면 간단한 확인 절차가 있습니다

만약 같은 기기에서 처음 접속하거나, 기기를 바꿨다면 고팡 인증 화면이 나타납니다. 4단어 시드를 입력하면 바로 신원이 확인되고, 서비스를 이용할 수 있습니다.

인증 레벨: 보안의 4단계

모든 서비스가 동일한 수준의 보안을 요구하지는 않습니다. 게시판 글을 읽는 것과, 범죄를 신고하거나 법적 서류를 제출하는 것은 다릅니다. 고팡은 이를 4단계로 구분합니다.

레벨 인증 방법 허용 기능 예시
L0 기기 자동 인식 일반 조회, AI 상담
L1 L0 + 얼굴 인증 신고 접수, 출동 승인, 소송 패키지 생성
L2 L1 + 지문(WebAuthn) 진료 기록 열람, 금융 거래
L3 L2 + 4단어 시드 계정 복원, 최고 보안 기능

파일과 함수로 보는
정확한 작동 원리

고팡 인증 시스템은 3개의 핵심 파일로 구성됩니다. 이 파일들은 모두 gopang.net에만 존재합니다. 각 하위 시스템은 자체 인증 파일을 두지 않으며, HTML에 단 한 줄만 삽입합니다.

gopang.net/auth/ ← 인증 관련 파일은 모두 여기에만 존재
  ├── gopang-sso.js ← 중앙 인증 라이브러리 (핵심)
  ├── subsystem-auth.js ← 하위 시스템 공용 인증 모듈
  └── silent-auth.html ← 크로스도메인 토큰 발급기
 
각 하위 시스템/ ← auth.js 파일 없음
  └── *.html ← script 태그 한 줄만 삽입

1단계 — 하위 시스템은 HTML 한 줄만 삽입한다

수백 개의 하위 시스템 각각에 인증 파일을 두면, 인증 로직이 변경될 때마다 수백 개를 모두 수정해야 합니다. 고팡은 이 문제를 근본적으로 해결합니다. 하위 시스템은 HTML에 script 태그 한 줄만 추가하면 됩니다. 인증 로직 전체가 gopang.net에서 원격으로 실행됩니다.

police.html / school.html / klaw.html / … (모든 하위 시스템 동일) HTML 
<!-- 고팡 인증 — 이 한 줄이 전부입니다 -->
<script type="module"
  src="https://gopang.net/auth/subsystem-auth.js">
</script>

더 세밀한 제어가 필요할 때는 함수를 직접 import해서 사용할 수도 있습니다.

고급 사용 예시 (선택적) JavaScript 
import { initAuth, requireLevel }
  from 'https://gopang.net/auth/subsystem-auth.js';

// L0 기본 인증
const user = await initAuth();

// 중요 기능(신고 접수, 소송 생성 등) 전 L1 요청
const ok = await requireLevel('L1');
if (ok) { /* 기능 실행 */ }
💡
핵심: 인증 로직을 수정할 일이 생기면 gopang.net/auth/subsystem-auth.js 하나만 수정합니다. 수백 개의 하위 시스템 전체에 즉시 반영됩니다. 각 하위 시스템은 아무것도 변경할 필요가 없습니다.

2단계 — gopang-sso.js가 4가지 경로를 순서대로 시도한다

gopangAuth.require('L0')가 호출되면, gopang-sso.js는 아래 4가지 경로를 빠른 것부터 순서대로 시도합니다. 하나라도 성공하면 즉시 인증을 완료합니다.

①
GWP (Gopang Widget Portal) 토큰 확인

고팡 앱 안에서 하위 서비스를 열 때, URL에 ?gwp=1&gwp_token=... 파라미터가 붙어 옵니다. _tryGwpToken() 함수가 이를 파싱하고 HMAC-SHA256으로 서명을 검증합니다.

가장 빠름 · GWP(Gopang Widget Portal) 내부 접속 시
②
세션 캐시 확인

이미 인증한 적이 있다면, sessionStorage['gopang_sso_token']에 1시간짜리 토큰이 저장되어 있습니다. _trySession() 함수가 이를 읽어 만료 여부를 확인합니다.

캐시 활용 · 같은 탭 재방문 시
③
로컬 기기 확인

_tryLocalStore() 함수가 같은 브라우저의 localStorage['gopang_user_v3']를 읽고, 현재 기기의 핑거프린트(_buildDeviceFingerprint())와 대조합니다. 일치하면 토큰을 발급합니다.

직접 접속 · 같은 기기일 때
④
Silent iframe (크로스도메인 해결)

_trySilentIframe() 함수가 gopang.net/auth/silent-auth.html을 1×1px 보이지 않는 iframe으로 삽입합니다. 이 iframe은 gopang.net 도메인에서 실행되므로 gopang.net의 localStorage를 직접 읽을 수 있습니다. 토큰 발급 후 postMessage로 결과를 전달합니다.

크로스도메인 해결 · 핵심 기술
⑤
리다이렉트 (최후 수단)

위 모든 경로가 실패하면 _redirectToGopang()이 호출됩니다. 사용자를 gopang.net/auth/silent-auth.html?return=현재URL로 보내 등록 또는 복원 UI를 보여줍니다. 완료 후 원래 서비스로 자동 복귀합니다.

3단계 — silent-auth.html이 크로스도메인을 해결한다

브라우저 보안 정책상 서로 다른 도메인의 localStorage는 읽을 수 없습니다. police.gopang.net은 gopang.net의 저장 데이터를 직접 읽지 못합니다. 이 문제를 silent-auth.html이 해결합니다.

🪟 유리창 너머 쪽지 전달에 비유하면:
A 방(police.gopang.net)에서는 B 방(gopang.net)의 서랍을 직접 열 수 없습니다. 하지만 B 방 안에 대리인(silent-auth.html)을 잠깐 들여보내면, 대리인이 서랍을 열어 내용을 확인하고 쪽지(postMessage)로 결과를 창문 너머로 건네줄 수 있습니다.

gopang.net/auth/silent-auth.html (핵심 부분) JavaScript 
// silent-auth.html은 gopang.net 도메인에서 실행됨
// → gopang.net의 localStorage를 직접 읽을 수 있음

async function _runSilentMode() {
  const user = _getLocalUser();  // localStorage['gopang_user_v3'] 읽기

  // 기기 핑거프린트 일치 확인
  const fpHex = await _buildDeviceFingerprint();
  if (user.fpHex !== fpHex) {
    _postToParent(null, 'DEVICE_CHANGED');
    return;
  }

  // 토큰 발급 후 postMessage로 전달
  const tokenObj = await issueToken(user, SERVICE);
  _postToParent(tokenObj, null);
}

function _postToParent(token, error) {
  // police.gopang.net으로 결과 전송
  window.parent.postMessage({
    type:  'GOPANG_SSO_TOKEN',
    token: token,
    error: error,
  }, targetOrigin);
}

4단계 — 기기 정체성: 어떻게 "나"를 증명하는가

고팡에는 아이디와 비밀번호가 없습니다. 대신 기기 핑거프린트와 4단어 시드로 신원을 증명합니다.

gopang-sso.js · _buildDeviceFingerprint() JavaScript 
async function _buildDeviceFingerprint() {
  // 기기 고유 정보 8가지를 조합
  const raw = [
    navigator.userAgent,          // 브라우저 종류
    navigator.language,           // 언어 설정
    screen.width + 'x' + screen.height, // 화면 크기
    screen.colorDepth,            // 색상 깊이
    Intl.DateTimeFormat().resolvedOptions().timeZone, // 시간대
    navigator.hardwareConcurrency, // CPU 코어 수
    navigator.deviceMemory,       // RAM 용량
    screen.pixelDepth,            // 픽셀 깊이
  ].join('|');

  // SHA-256 해시 → 64자리 16진수 문자열
  const buf = await crypto.subtle.digest('SHA-256', new TextEncoder().encode(raw));
  return Array.from(new Uint8Array(buf))
    .map(b => b.toString(16).padStart(2, '0')).join('');
}

이 핑거프린트를 앞 16바이트만 사용해 IPv6 형식의 고유 주소로 변환합니다. 이것이 고팡에서 사용자를 식별하는 GUID(고유 식별자)가 됩니다. 예: 2601:db80:a3f2:...

Part 3  ·  더욱 강력한 보안

OpenHash 기반 PDV
— 인증을 넘어선 증거

지금까지 설명한 SSO 인증은 "이 사람이 누구인가"를 증명합니다. 그런데 고팡은 여기서 한 걸음 더 나아갑니다. 인증 행위 자체를 포함한 모든 데이터 접근 이력을 수학적으로 변조 불가능한 형태로 기록합니다. 이것이 PDV(Private Data Vault, 개인 데이터 금고)와 OpenHash의 역할입니다.

🏦 공증된 금고에 비유하면:
일반 인증은 금고 열쇠를 확인하는 것입니다. 고팡의 PDV+OpenHash는 "누가, 언제, 어떤 열쇠로, 어느 서랍을 열었는지"를 공증 기관이 실시간으로 기록하는 것과 같습니다. 이 기록은 금고 주인도, 고팡도, 심지어 법원도 사후에 변경할 수 없습니다. 기록 자체가 법적 증거가 됩니다.

PDV란 무엇인가

PDV(Private Data Vault)는 사용자의 모든 데이터를 사용자 본인의 기기에 암호화하여 저장하는 개인 데이터 금고입니다. 인증 이력, 서비스 이용 기록, 신고 내역, 진료 기록 등이 모두 PDV에 저장됩니다. 고팡 서버도, 하위 시스템도, 정부 기관도 사용자의 동의 없이 내용을 볼 수 없습니다.

PDV의 각 레코드는 6하원칙(Who·When·Where·What·How·Why) 구조로 기록됩니다.

PDV 레코드 구조 예시 — K-Police 신고 접수 JSON 
{
  "svc":   "police",
  "type":  "incident_report",
  "who":   "2601:db80:a3f2:…",        // 신고자 IPv6 GUID
  "when":  "2026-06-01T22:15:00Z",
  "where": "police.gopang.net",
  "what":  { "crime_type": "강도", "risk_level": "CRITICAL" },
  "how":   { "auth_level": "L1", "method": "gopang-sso" },
  "why":   { "goal": "범죄 신고 및 출동 요청" },
  "openhash": "a3f2b8c1…"          // OpenHash 앵커 해시값
}

OpenHash란 무엇인가

OpenHash는 고팡의 핵심 기술로, PHLD(Probabilistic Hierarchical Ledger Distribution, 확률적 계층 분산 원장) 방식으로 동작하는 분산 해시 네트워크입니다. 데이터가 생성되는 순간 그 내용의 해시값이 전 세계에 분산된 노드에 즉시 등록됩니다. 한 번 등록된 해시는 수학적으로 변경이 불가능합니다.

📰 신문 기사에 비유하면:
오늘 아침 신문에 실린 기사는 수백만 부가 배포된 뒤에는 아무도 내용을 바꿀 수 없습니다. OpenHash는 이와 같습니다. 단, 신문은 수백만 부를 인쇄해야 하지만, OpenHash는 해시값(지문) 하나만으로 전 세계 노드에 즉시 "공증"합니다.

PDV + OpenHash가 인증에 더하는 것

기존 SSO 인증은 "지금 이 순간"의 신원만 확인합니다. PDV + OpenHash는 여기에 "과거의 모든 순간"에 대한 불변 증거를 더합니다.

🔐
인증 이력 불변 기록

사용자가 서비스에 로그인할 때마다 auth_level, ipv6, 타임스탬프가 PDV에 기록되고 OpenHash로 앵커링됩니다. "내가 언제 로그인했는가"가 법적 증거가 됩니다.

위조 불가 · 수학적 보증
🛡️
발신자 서명 (부인 방지)

사용자의 seedHex로 HMAC-SHA256 서명된 모든 행위(신고, 계약, 진술)는 나중에 "나는 그런 적 없다"고 부인할 수 없습니다. 서명은 사용자의 4단어 시드에서만 생성되며, 고팡 서버도 복제할 수 없습니다.

부인 방지 · Non-repudiation
⛓️
해시 체인 연속성

각 PDV 레코드는 이전 레코드의 해시를 포함합니다. 중간 기록을 삭제하거나 변조하면 체인이 깨져 즉시 탐지됩니다. 블록체인과 동일한 원리이지만, 개인 기기에서 동작하므로 프라이버시가 보호됩니다.

변조 즉시 탐지 · 체인 무결성
🌐
분산 앵커링 (단일 장애점 없음)

OpenHash 노드는 전 세계에 분산되어 있습니다. 고팡 서버가 해킹되거나 서비스가 중단되어도, 이미 앵커링된 해시는 전 세계 노드에 살아 있습니다. 어느 한 기관도 기록을 지울 수 없습니다.

고가용성 · 탈중앙화

  │
  ↓

  │  해시값 → PHLD 분산 원장에 즉시 등록

   "누가·언제·어디서·무엇을·어떻게·왜" 변조 불가 기록
⚖️
법적 효력: PDV + OpenHash의 결합이 만드는 것은 자기완결 증거 구조입니다. 피해자가 고팡을 통해 신고하거나, 계약을 체결하거나, 진술을 남기면, 그 순간 OpenHash가 내용과 시각을 불변으로 기록합니다. 상대방이 자신의 기기에서 메시지를 삭제해도, 피해자의 PDV와 OpenHash 앵커가 증거로 남습니다. 별도의 공증이나 법원 제출 없이도 증거 효력을 갖습니다.

PDV는 일상을 기록한다 — 그래서 거짓말을 탐지한다

PDV는 단순한 로그 파일이 아닙니다. 사용자의 일상과 업무 전체를 6하원칙에 따라 연속적으로 기록합니다. 사용자가 어디에 있었는지, 몇 시에 어떤 서비스에 접속했는지, 어떤 행위를 했는지가 모두 시계열로 쌓입니다.

이 기록은 오직 진실만을 반영합니다. 기기 핑거프린트, GPS 위치, 접속 시각은 사용자가 임의로 조작할 수 없기 때문입니다. 따라서 PDV 기록과 어긋나는 인증 시도는 즉시 이상 징후(Abnormal Login Attempt)로 식별됩니다.

🗓️ 일기장에 비유하면:
매일 빠짐없이 기록된 일기장이 있다면, "나는 그날 서울에 있었다"는 거짓 주장은 일기장이 반박합니다. PDV는 변조 불가능한 디지털 일기장입니다. 평소와 다른 시간대, 다른 나라, 다른 기기에서 로그인이 시도되면, 일기장과 맞지 않는 이 사건은 즉시 의심 대상이 됩니다.

구체적으로 다음과 같은 이상 징후가 탐지됩니다.

📍
위치 이상 (Impossible Travel)

PDV에 기록된 직전 접속 위치가 제주도인데, 10분 후 모스크바에서 로그인이 시도됩니다. 물리적으로 불가능한 이동입니다. 고팡은 이를 즉시 비정상으로 판정하고 해당 세션을 차단합니다.

위치 × 시간 교차 분석
🕐
시간대 이상 (Temporal Anomaly)

사용자의 PDV 기록상 평소 접속 시간대가 오전 8시~오후 11시인데, 새벽 3시에 L2 레벨의 금융 거래가 시도됩니다. 패턴과 다른 시간대의 고위험 행위는 자동으로 추가 인증을 요구합니다.

행동 패턴 기반 탐지
📱
기기 불일치 (Device Mismatch)

기기 핑거프린트가 평소와 다릅니다. 새 기기에서 접속하면 4단어 시드 복원 절차를 요구합니다. 시드 없이 반복 시도하면 해당 IPv6 계정을 임시 잠금하고 사용자에게 경고를 발송합니다.

기기 핑거프린트 연속 검증
🔁
반복 실패 (Brute Force)

짧은 시간 안에 동일 계정에 대한 인증 실패가 반복됩니다. 고팡은 실패 횟수와 간격을 PDV에 기록하고, 임계값 초과 시 자동으로 접근을 차단하며 security.gopang.net에 즉시 경보를 발송합니다.

자동 차단 · 경보 발송

security.gopang.net
— 나만을 위한 개인 경호원

기존의 보안 시스템은 중앙 서버가 수백만 명의 데이터를 한꺼번에 감시합니다. 은행의 이상거래탐지 시스템, 포털의 로그인 보안 알림 — 모두 타인(기업·정부)이 나의 활동을 들여다보는 구조입니다. 내 데이터는 그들의 서버에 있고, 그들이 판단합니다.

security.gopang.net은 근본적으로 다릅니다. 이 시스템은 수백만 명을 동시에 감시하는 관제탑이 아닙니다. 단 한 명의 사용자 — 바로 나 — 의 PDV와 인증 과정만을 전담으로 관찰합니다. 나의 데이터는 내 기기에 있고, 내 PDV가 판단하며, 이상이 생기면 오직 나에게만 통보합니다. 타인은 관여하지 않습니다.

🛡️ 건물 경비원 vs 개인 경호원:
기존 보안 시스템은 건물 경비원입니다. CCTV로 건물 안 모든 사람을 지켜봅니다. 내 행동도 그들의 모니터에 찍힙니다. 반면 security.gopang.net은 내 전담 개인 경호원입니다. 오직 나만을 24시간 관찰하고, 위협이 감지되면 나에게만 보고합니다. 경비원은 내 사생활을 볼 수 있지만, 개인 경호원은 오직 나의 명령만 따릅니다.

이상 징후 감지 시
4단계 대응 절차

security.gopang.net이 나의 PDV에서 이상 징후를 감지하면, 외부 기관에 알리는 것이 아니라 나에게 직접 추가 인증을 요구합니다. 위협의 심각도에 따라 4단계가 순서대로 적용됩니다. 각 단계를 통과하면 이전 단계보다 강력한 신원 확인이 완료된 것입니다.

1단계
얼굴 인증 (Face)

카메라로 얼굴을 촬영합니다. 등록 시 저장된 얼굴 벡터와 대조합니다. 얼굴 데이터는 내 기기 밖으로 전송되지 않습니다. 경미한 이상 징후(새 위치, 새 시간대)에 적용됩니다.

L1 인증 · 기기 내 처리
2단계
지문 인증 (Fingerprint · WebAuthn)

스마트폰 또는 컴퓨터의 생체 인증(지문·Face ID)을 통해 WebAuthn 방식으로 신원을 확인합니다. 피싱이나 원격 해킹으로는 절대 통과할 수 없습니다. 기기를 물리적으로 소지하고 있어야 합니다.

L2 인증 · WebAuthn 표준
3단계
성문 인증 (Voice Print)

마이크로 목소리를 녹음합니다. 등록된 성문(음성 지문)과 대조합니다. 얼굴·지문과 달리 원격 전화 통화 상황에서도 적용 가능합니다. 심각한 이상 징후(불가능한 위치 이동, 반복 실패)에 적용됩니다.

L2+ 인증 · 원격 적용 가능
4단계
오프라인 인증 (Offline Verification)

관공서·금융 기관을 직접 방문하거나, 4단어 시드를 입력하여 신원을 최종 확인합니다. 디지털 인증이 모두 실패했거나 계정이 완전히 잠긴 최고 위험 상황에 적용됩니다. 이 단계를 통과하면 계정이 완전히 복원됩니다.
