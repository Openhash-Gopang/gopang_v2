## 1. 고팡이란 무엇인가

### 기술적 내용

고팡은 ULA 기반 IPv6 라우팅, DHT 이름 해석, Stealth Address를 결합한 탈중앙 분산 메시징 네트워크입니다. 모든 데이터는 중앙 서버 대신 사용자 기기의 PDV(Personal Data Vault)에 암호화되어 저장됩니다. 모든 소통의 해시는 OpenHash 분산원장에 즉시 등록됩니다.

### 법률가를 위한 해설

**기존 메신저(카카오톡 등):** 모든 대화가 회사 서버에 저장됩니다. 회사가 접근할 수 있고, 수사기관이 영장을 집행하면 서버 전체를 압수할 수 있습니다.

**고팡:** 대화 내용이 각 사용자의 스마트폰에만 있습니다. 중간 서버(L1 노드)는 "언제, 어느 주소 간에 메시지가 오갔다"는 배송 기록과 메시지가 저장된 위치(해시값)만 가집니다. 내용은 없습니다.

**법적 의미:** 수사기관이 L1 노드에 영장을 집행해도 메시지 내용을 얻을 수 없습니다. 내용에 접근하려면 사용자 기기 자체에 대한 별도 법적 절차가 필요합니다.

**고팡이 현행 사법 체계에 대해 취하는 입장:** 경쟁이나 대체가 아니라 보완입니다. 법률 서비스 접근성을 높이고, 기술적 수단으로 법적 절차의 신뢰성을 보강하는 역할입니다.

## 2. 고팡의 두 기술 토대: PDV와 OpenHash

### 2.1 PDV (Personal Data Vault)

### 기술적 내용

```
저장 위치: 사용자 디바이스 (스마트폰, 컴퓨터)
암호화: AES-256, 개인키로만 복호화 가능
접근 권한: 사용자 본인 + 본인 AI 비서만
L1 노드가 보관하는 것: 라우팅 메타데이터, CID (내용 위치 정보)
L1 노드가 보관하지 않는 것: 메시지 내용, 개인 식별 정보
```

### 법률가를 위한 해설

**비유:** 모든 편지를 우체국 창고에 보관하는 것(기존 메신저)이 아니라, 각 수신자의 집 금고에 보관하는 것(고팡 PDV)입니다. 우체부(L1 노드)는 편지를 배달했다는 기록만 갖고, 내용은 보지 않습니다.

**법적 의미:** 수사기관이 우체국(L1 노드)에 영장을 내도 편지 내용을 얻을 수 없습니다. 각 사람의 집 금고(기기)에 대한 별도 영장이 필요합니다.

**증거 보전 관점:** 소통은 쌍방향입니다. A가 B에게 보낸 메시지는 A의 PDV에도, **B의 PDV에도** 저장됩니다. 그리고 A의 디지털 서명이 첨부됩니다. 이것이 다음 섹션에서 설명할 자기완결 증거 구조의 핵심입니다.

### 2.2 OpenHash

### 기술적 내용

```
OpenHash 해시 체인:
h_i = SHA-256(h_{i-1} ∥ data_i ∥ block_height)

모든 메시지 발송 즉시:
  content_hash = SHA-256(메시지_내용)
  → 물리적 계층 분산 네트워크에 등록 (읍면동→시군구→광역→국가→글로벌)
  → 이후 chain에 포함, 소급 변경 수학적 불가능

에너지 소비: PoW 대비 0.01% 미만
처리량: 단일 노드 4,399 TPS (단일 머신 환경 실측)
```

### 법률가를 위한 해설

**비유:** 공증인이 문서에 날인하는 것과 유사하지만, 공증인이 단 한 명인 것과 달리 OpenHash는 전국·전 세계에 분산된 수천 개의 노드가 동시에 기록합니다. 한 노드가 훼손되어도 나머지가 원본을 보존합니다.

**OpenHash가 보증하는 것:**
- "이 메시지가 이 시각에 이 내용으로 존재했다"
- 발송 시각 (블록 높이로 확정, 조작 불가)
- 내용 무결성 (해시 불일치 시 변조 즉시 감지)

**법원이 할 수 있는 것:** OpenHash 네트워크에 메시지 해시를 조회하면, 고팡 회사와 무관하게 독립적으로 해당 메시지의 존재·시각·무결성을 확인할 수 있습니다.

**기존 블록체인과의 차이:** 비트코인·이더리움 같은 에너지 집약적 합의 메커니즘을 사용하지 않습니다. 물리적 계층 인프라(통신망)의 신뢰를 활용하여 합의 없이 무결성을 보장합니다. 모든 소통을 실시간 기록하는 것이 경제적으로 실현 가능합니다.

## 3. 자기완결 증거 구조: Legal Hold가 불필요한 이유

### 기술적 내용

고팡의 증거 구조는 세 요소의 조합으로 완결됩니다.

```
① 피해자 PDV 원본
   소통은 쌍방향 → 발신자·수신자 양측 PDV에 동일 메시지 저장
   가해자가 자신의 PDV에서 삭제해도 피해자 PDV는 영향 없음

② 발신자 ECC 디지털 서명
   모든 메시지에 발신자 서명 자동 첨부
   서명 검증식: s·G =?= R + c·pk (Schnorr 기반)
   피해자 PDV의 원본에 서명 포함 → 발신 사실 수학적 부인 불가

③ OpenHash 해시 체인
   발송 즉시 SHA-256(메시지_내용) → OpenHash 등록
   h_i = SHA-256(h_{i-1} ∥ content_hash ∥ block_height)
   시각·내용 소급 변경 수학적 불가능
```

v1.2까지 존재했던 Legal Hold(스마트 컨트랙트 기반 삭제 금지 메커니즘)는 이 구조에서 불필요합니다. 가해자가 자신의 데이터를 삭제해도 증거가 사라지지 않기 때문입니다.

### 법률가를 위한 해설

**왜 Legal Hold가 필요했는가 (v1.2까지):**

기존 메신저에서 증거 보전 신청이 필요한 이유는 "가해자가 회사 서버 측 관리자에게 데이터 삭제를 요청할 수 있기 때문"이었습니다. 법원이 개입하기 전에 서버 측 데이터가 사라질 위험이 있었습니다.

**왜 고팡에서는 이 전제가 성립하지 않는가:**

고팡에서 데이터는 각 사용자의 기기에 있습니다. A(가해자)가 자신의 기기에서 메시지를 지워도, B(피해자)의 기기에 있는 동일 메시지는 전혀 영향받지 않습니다. 서버 관리자가 없으니 서버 측 삭제 자체가 불가능합니다.

**세 요소의 역할을 일상 언어로:**

```
① 피해자 PDV = "피해자가 원본을 갖고 있다"
② 발신자 서명 = "그 원본에 가해자의 지문이 찍혀 있다"
③ OpenHash   = "공증인이 그 지문이 언제 찍혔는지 기록했다"
```

이 세 가지가 있으면 가해자가 무엇을 지워도, 무슨 주장을 해도, 증거가 흔들리지 않습니다.

**어떤 시나리오에도 작동합니다:**

| 시나리오 | 대응 |
|---------|------|
| 가해자가 자신의 PDV에서 삭제 | 피해자 PDV에 원본 있음 |
| "내가 보낸 게 아니다" | 원본에 가해자 서명 — 수학적 부인 불가 |
| "내용이 바뀌었다" | OpenHash 해시로 즉시 검증 |
| "날짜가 다르다" | OpenHash 블록 높이로 발송 시각 확정 |
| 피해자가 내용을 조작하려 시도 | 가해자 서명 없이 위조 불가 |

**수신자가 없거나 고팡 비사용자인 경우:** 이 케이스는 법적 분쟁으로 이어지는 빈도가 극히 낮습니다. 설령 분쟁이 발생해도 현행 전자서명법·전자문서법의 일반 원칙으로 처리하는 것이 비용 대비 효용 측면에서 합리적입니다. 고팡은 이를 위한 별도 시스템을 구성하지 않습니다.

## 4. 영지식 증명(ZKP): 비밀을 공개하지 않고 사실을 증명하는 기술

### 기술적 내용

고팡은 Schnorr 프로토콜 기반 ZKP로 메시지 발신자의 주소 소유권을 증명합니다.

```
파라미터:
  G  : 타원곡선 생성자 (Curve25519)
  sk : 사용자 개인키 (비공개)
  pk : 사용자 공개키 = sk·G (공개)

프로토콜:
  [1] k ← rand, R = k·G → 법원에 전송
  [2] 법원: c ← rand → 사용자에게 전송
  [3] s = (k + c·sk) mod n → 법원에 전송
  [4] 법원 검증: s·G =?= R + c·pk ✓

비대화형 (파일 제출용):
  c = H(pk || R || H_m || Nonce || timestamp)
  π = (R, s, c, H_m, Nonce, timestamp)
```

법원 Verification API 수수료: 0.01 GDC / 1회.

### 법률가를 위한 해설

**비유:** 금고의 비밀번호를 알고 있다는 것을 증명하되, 비밀번호 자체를 말하지 않고 금고를 열어 보이는 것입니다. 수학적으로 더 엄밀한 방식입니다.

**증명할 수 있는 것:**
- "이 메시지는 이 주소에서 발송되었고, 나는 그 주소의 소유자다"
- 법원은 제출받은 수치 세 개(R, s, c)와 검증식으로 독립 확인

**증명할 수 없는 것:**
- 메시지 내용, 다른 통신 내역, 사용자의 실제 이름·주민번호

**v1.3에서의 ZKP 위치:** ZKP는 PDV+OpenHash 자기완결 증거 구조를 *보완*하는 추가 도구입니다. 피해자 PDV의 원본과 발신자 서명만으로도 핵심 증거는 완결됩니다. ZKP는 계정 소유권을 별도로 확인하고 싶을 때, 또는 증명의 무게(proof_weight)를 통해 신뢰도를 자발적으로 높이고 싶을 때 사용합니다.

**비대화형 증명(파일 제출):** 법원과 실시간 대화 없이 증명서(π)를 파일로 생성해 제출할 수 있습니다. 공증받은 진술서와 유사하게 취급될 수 있습니다.

## 5. 타임스탬프 신뢰성: 발신 시각의 법적 증명 가능성

### 기술적 내용

```
T_device : 발신자 PDV 로컬 시계 (사용자가 조작 가능)
T_l1     : L1 중계 노드 3개의 수신 시각 중앙값 (NTP ±500ms)
T_anchor : OpenHash 블록 타임스탬프 (블록 높이로 확정, 변경 불가)

신뢰 우선순위: T_anchor > T_l1 > T_device

검증:
  |T_anchor - T_l1| > 30초 → Suspicious
  T_device > T_l1 + 5초   → Flagged
  모두 통과               → Valid
```

### 법률가를 위한 해설

**왜 타임스탬프가 중요한가:** "이 메시지가 언제 발송되었는가"는 분쟁에서 핵심 쟁점입니다. 기존 스마트폰의 경우 기기 시계를 변경하면 타임스탬프를 조작할 수 있습니다.

**고팡의 3중 타임스탬프:**

- **기기 시계(T_device):** 발신자 스마트폰 시계. 조작 가능. 단독으로는 신뢰하기 어렵습니다.
- **중계 서버 기록(T_l1):** 메시지가 중계 서버 세 곳을 통과할 때 각 서버가 기록한 수신 시각의 중앙값. 서버들은 인터넷 표준 시각(NTP)에 동기화되어 개인이 조작하기 어렵습니다.
- **블록체인 기록(T_anchor):** 가장 신뢰도가 높습니다. OpenHash 블록에 기록된 후 변경 불가능합니다. 법원·상대방·제3자가 고팡과 무관하게 독립 조회·검증 가능합니다.

세 값의 편차가 크면(예: 기기 시계가 서버 기록보다 5초 이상 앞서 있으면) 시스템이 자동으로 의심 신호를 표시합니다. 증거 일관성 보고서에 이 편차가 수치로 포함되므로, 법원이 타임스탬프 신뢰도를 구체적으로 판단할 수 있습니다.

## 6. 법원 전용 검증 시스템 (v1.3 개정)

### 기술적 내용

```
기본 URL: https://court-api.gopang.net/v1/
인증: 법원 공개키로 서명된 JWT, TLS 1.3 양방향 인증서
수수료: 0.01 GDC / 1회 검증

엔드포인트 (v1.3):
  POST /verify/ownership       → ZKP 소유권 검증
  POST /verify/signature       → 발신자 디지털 서명 검증 (신규)
  GET  /evidence-report/{id}   → 증거 일관성 보고서
  GET  /openhash/verify/{hash} → OpenHash 해시 독립 검증 (신규)
  GET  /proof-weight/{id}      → 증명 무게 조회
```

*제거된 엔드포인트: `/legal-hold/apply`, `/legal-hold/release` (Legal Hold 자체 제거)*

**증거 일관성 보고서 포함 항목 (v1.3):**

```
1. 메시지 식별 정보    : 고유번호, 저장 위치(CID), 발신자 주소
2. 타임스탬프         : T_device, T_l1(중앙값), T_anchor, 편차
3. 시각 신뢰도 판정   : Valid / Warning / Suspicious / Flagged
4. OpenHash 검증     : 해시 체인 재계산 결과 (변조 여부)
5. 발신자 서명 검증   : 디지털 서명 유효성 (신규)
6. ZKP 소유권 증명   : π = (R, s, c, H_m) (선택적)
7. 증명 무게         : 발신자 소각 GDC 금액 (선택적)
8. 종합 일관성 점수   : 0~1 (참고값)
```

### 법률가를 위한 해설

**이 시스템이 법원에 제공하는 것:** 고팡에 문의하지 않고, 법원이 스스로 디지털 증거의 진정성을 독립적으로 확인할 수 있는 창구입니다.

**v1.3에서 추가된 두 엔드포인트:**

`POST /verify/signature`: "이 메시지에 첨부된 서명이 정말 이 주소의 소유자가 만든 것인가"를 직접 검증합니다. 피해자 PDV의 원본을 제출하면, 법원이 서명의 수학적 유효성을 확인합니다.

`GET /openhash/verify/{hash}`: "이 해시값이 OpenHash 네트워크에 언제 등록되었는가"를 조회합니다. 고팡 서버와 무관하게 분산 네트워크에서 직접 확인합니다.

**실제 소송에서의 활용 흐름 (v1.3):**

```
[1] 피해자가 자신의 PDV에서 문제 메시지 추출
    → 원본 + 발신자(가해자) 디지털 서명 포함

[2] 법원이 서명 검증
    POST /verify/signature
    → "이 서명이 피고 계정의 것" 수학적 확인

[3] 법원이 OpenHash에서 시각 독립 확인
    GET /openhash/verify/{content_hash}
    → 발송 시각 확정, 내용 변조 없음 확인

[4] 증거 채택
    → 피고는 자신의 서명이 첨부된 메시지에 대해 발송 부인 불가
```

**접근 권한:** 법원 인증서 보유 기관만 접근 가능. 수사기관은 반드시 법원 경유. 모든 조회 기록이 OpenHash에 불변 기록됩니다.

## 7. K-Law AI: 법적 보조 도구로서의 위치와 한계

### 7.1 현재 성능 (2026년 5월 기준)

K-Law AI는 대법원 실제 판결과의 일치율 **73%**를 달성했습니다(내부 측정, 독립 제3자 검증 미완료). 측정 대상은 대법원이 사법공보에 선별 공개한 300건으로, 법리 판단이 가장 어렵고 불확실성이 높은 사건들입니다.

전체 소송으로 확장하면 87~92%로 상향 추정됩니다〔추정치, ±5%p, 실측 미완료〕.

### 7.2 K-Law의 법적 성격

| 구분 | 내용 |
|------|------|
| **법적 구속력** | 없음. 당사자 간 자율 합의의 기초 자료 |
| **강제 집행** | 불가. 단, 당사자가 스마트 컨트랙트에 사전 동의한 경우 자동 이행 |
| **법률 자문 대체** | 불가. 법률 정보 제공 및 위험 진단 도구 |
| **사법 판결 대체** | 불가. AI는 사법권 행사 불가 (현행 법령) |

### 7.3 인간 법관과의 상보적 관계

K-Law는 **예방법학 도구**로 포지셔닝됩니다. 분쟁 발생 전 위험성을 진단하고, 당사자들이 분쟁을 회피할 수 있도록 돕는 역할입니다.

- 계약 체결 전 법적 위험 진단
- 소송 전 판결 예측 (참고용)
- 분쟁 당사자 간 자율 중재 지원
- 변호사의 판례 검색·계약서 검토 효율화

**PDV+OpenHash와의 연동:** K-Law는 고팡에서 보전된 PDV 원본 증거를 분석 입력으로 사용할 수 있습니다. OpenHash가 무결성을 보증한 증거 위에서 K-Law가 법리를 분석하므로, 사실 다툼 없이 법리 논쟁에 집중할 수 있습니다.

## 8. 개인정보 보호 구조와 적법한 법집행 협조

### 기술적 내용

```
L1 노드가 보관하는 것:  CID, 라우팅 타임스탬프, 발신·수신 주소
L1 노드가 보관하지 않는 것: 메시지 내용, 사용자 실제 신원 정보
메시지 내용: IPFS 분산 저장소에 암호화 저장, 복호화 키는 수신자 PDV에만 존재
```

### 법률가를 위한 해설

**수사기관 관점의 고팡 구조:**

**L1 노드에 영장을 집행하면:** "언제, 어느 주소 간에 메시지가 오갔다"는 배송 기록과 메시지가 저장된 위치 정보를 얻을 수 있습니다. 내용은 없습니다.

**피해자가 증거를 제출하면:** 피해자의 PDV에서 원본 + 발신자 서명을 꺼낼 수 있습니다. 법원이 Verification API로 서명과 OpenHash를 검증합니다.

**발신자 기기를 압수하면:** PDV에 해당 기기로 주고받은 메시지가 있습니다. 기기 암호화 설정에 따라 추가 해제 절차가 필요합니다.

**고팡이 설계에 포함한 법집행 협조 채널:**
- `POST /verify/signature` — 발신자 서명 검증
- `GET /openhash/verify/{hash}` — OpenHash 해시 독립 검증
- `POST /verify/ownership` — ZKP 소유권 확인
- `GET /evidence-report/{id}` — 증거 일관성 보고서

**Legal Hold가 없는 것에 대하여:** v1.3부터 Legal Hold를 제거했습니다. 가해자가 자신의 PDV에서 메시지를 삭제해도, 피해자 PDV의 원본과 OpenHash 해시가 여전히 존재합니다. 법원 입장에서 Legal Hold보다 강력한 증거 보전 효과가 구조 자체에 내장되어 있습니다.

| 관점 | 분석 |
|------|------|

- 옵션 A: 네트워크 내 기술적 복구만 (법적 효력 미보장)
- 옵션 B: 공증인 앞에서 '복구 위임장' 작성 후 GAS에 해시 등록
- 옵션 C: 유언장에 GAS 복구 절차 명시

**복구가 보장하지 않는 것:** 이전 개인키로 암호화된 메시지 복호화, 법적 소유권 이전, GDC 토큰 재산적 가치 이전.

**법무법인 검토 필요 사항:** 각국 전자서명법상 복구 절차의 효력, 사망 시 상속 절차와의 관계, 대리권 범위.

2. **OpenHash 해시 체인의 증거능력 검토:** 블록체인·분산원장 기반 타임스탬프의 각국 민사·형사 소송 증거능력. OpenHash가 전자서명법상 '공인전자문서보관소' 또는 유사 기관으로 인정받을 수 있는지 여부.

**중기 협력 (3~6개월):**

4. **K-Law 현지화 법적 검토:** K-Law의 법률 정보 제공 서비스가 각국 변호사법상 무허가 법률 자문에 해당하는지 여부.

6. **외환거래법 검토:** 다국적 통화 풀 운영이 각국 외환거래법상 외국환업무 취급에 해당하는지 여부와 필요 인·허가.

| 비교 쌍 | 일치도 | 표본 | 성격 |

### 11.3 73%가 정체된 이유: 입력 정보 부족

```
요약본(A4 1p) 입력 시 정확도: 73.4%  〔실측〕
원본 전문 입력 시 정확도:    93.5%  〔실측〕
정보 손실률(g값):            0.785  〔실측값에서 산출〕
```

현재 73%의 한계는 K-Law의 판단 능력이 아니라, 입력되는 사건 정보의 불완전성에서 비롯됩니다.

### 11.4 전체 소송 기준 추정

| 사건 계층 | 비중 | K-Law 예상 일치도 | 성격 |
|---------|------|----------------|------|

| Layer 2-A: 2심 확정 사건 | ~15% | 85~90% | 〔추정치〕 |

### 11.5 99.9% 이론적 상한에 대하여

모든 요건사실이 완전히 확정된다는 전제 하의 수학적 천장입니다. 세 조건이 요구됩니다: 법령·판례 불변, 모든 사실 완전 확정, 대법원 판단의 순수 논리·법리적 추론. 세 번째 조건은 법철학적으로 논쟁적입니다. **이것은 현실 사법 과정에 대한 묘사가 아닙니다.**

## 12. 주요 기술 용어 해설

법률 전문가가 이 자료를 이해하는 데 필요한 기술 용어를 법률적 언어와 비유로 설명합니다.

| 기술 용어 | 법률적 의미 |
|---------|-----------|
| **PDV (Personal Data Vault)** | 사용자 기기 내 암호화 개인 저장소. 본인의 집 금고. 회사도 접근권이 없습니다. |
| **OpenHash** | 분산 네트워크 기반 불변 기록 시스템. 공증인이 수천 명인 공증 시스템. 기록 후 소급 변경 불가. |
| **발신자 디지털 서명** | 모든 메시지에 자동 첨부되는 발신자의 수학적 지문. 서명 후 발송 사실 부인 불가. |
| **자기완결 증거 구조** | 피해자 PDV 원본 + 발신자 서명 + OpenHash 해시 체인의 3요소 조합. Legal Hold 없이 증거가 완결됩니다. |
| **해시(Hash)** | 문서의 디지털 지문. 단 한 글자라도 바뀌면 해시값이 완전히 달라집니다. |
| **블록체인 / OpenHash 체인** | 변경 불가능한 공개 장부. 기록 후 소급 수정이 수학적으로 불가능합니다. |
| **암호화 (E2EE)** | 봉인된 편지. 수신자만 열어볼 수 있으며 중간 배달부는 내용을 볼 수 없습니다. |
| **개인키** | 계정 소유권을 증명하는 비밀번호에 해당. 분실 시 복구가 어렵습니다. |
| **스마트 컨트랙트** | 조건 충족 시 자동 이행되는 계약 코드. 중간 개입 없이 계약이 자동 실행됩니다. |
| **L1 노드** | 메시지 중계 서버. 배송 기록(메타데이터)만 갖고 내용은 모릅니다. |
| **ZKP (영지식 증명)** | 비밀 없이 사실을 증명하는 수학적 기법. 금고를 열어 보이되 비밀번호는 말하지 않는 것. |
| **Trust Level** | 익명(L0) → 전화번호 인증(L1) → GDC 예치(L2) → KYC(L3)의 신원 인증 단계. |
| **CID (콘텐츠 식별자)** | 파일 내용 기반 고유번호. 내용이 바뀌면 CID도 바뀌므로 동일성 확인에 활용. |
| **GDC (Gopang Digital Currency)** | 고팡 네트워크 내 결제·인센티브·거버넌스 디지털 토큰. |
