## 1. 개요 및 설계 철학

고팡 주소 체계(GAS)는 ULA 기반 IPv6 라우팅, DHT 이름 해석, Stealth Address를 결합하여 **검색 가능성**과 **프라이버시**를 동시에 달성하는 탈중앙 주소 체계입니다.

### 1.1 고팡의 두 기술 토대: PDV와 OpenHash

고팡을 다른 모든 메시징 시스템과 구분하는 핵심은 두 가지 기술의 결합입니다.

**PDV(Personal Data Vault — 개인 데이터 볼트)**는 모든 소통 데이터를 사용자 본인의 기기에 암호화하여 저장합니다. 중앙 서버는 메시지 내용에 접근할 수 없습니다. 데이터의 소유권과 보관 책임이 온전히 사용자에게 있습니다.

**OpenHash**는 모든 소통이 발생하는 순간, 그 내용의 해시값을 물리적 계층 분산 네트워크에 즉시 등록합니다. 한 번 등록된 해시는 변경 불가능합니다. OpenHash는 "이 데이터가 이 시각에 이 내용으로 존재했다"는 사실을 수학적·분산적으로 보증합니다.

이 두 기술의 결합이 만들어내는 것이 고팡의 **자기완결 증거 구조**입니다(§20 참조).

### 1.2 v1.6의 핵심 전환: Legal Hold 제거 및 증거 체계 재정립

v1.4~v1.5에서 도입한 Legal Hold(증거 보존 모드)는 중앙 서버 기반 시스템의 결함—가해자가 서버 측 데이터를 삭제할 수 있다는 가정—을 보완하기 위한 장치였습니다.

그러나 PDV+OpenHash 구조에서 이 가정은 성립하지 않습니다.

```
가해자가 자신의 PDV에서 메시지를 삭제하더라도:
  ① 피해자의 PDV에 동일한 원본이 보관되어 있다
  ② 원본에는 발신자(가해자)의 디지털 서명이 첨부되어 있다
  ③ OpenHash 해시 체인이 해당 메시지의 존재와 내용을 불변으로 기록한다
```

세 요소의 조합으로 Legal Hold 없이도 증거 구조가 완결됩니다. v1.6은 Legal Hold를 전면 제거하고, PDV+OpenHash 기반의 자기완결 증거 체계를 고팡의 핵심 설계로 확립합니다.

### 1.3 GDC as Protocol Primitive (v1.5 계승)

GDC는 주소 생성, 암호화, 라우팅, 합의, 경제적 인센티브 전반에 내장된 프로토콜 자원입니다.

| 기존 분리 모델 | 통합 모델 (v1.6) |
|--------------|----------------|
| GDC는 수수료 지불용 외부 토큰 | GDC는 주소·암호화·라우팅·합의의 입력값 |
| 사용자는 GDC를 '보유'함 | GDC 상태가 공개키 해시의 일부 |
| 수수료는 메시지 전송 후 지불 | 수수료는 암호화 및 라우팅 경로 선택의 입력값 |
| Legal Hold로 증거 잠금 | PDV+OpenHash로 증거 구조 자체가 불변 |

### 1.4 핵심 명제

> **"GDC는 고팡 네트워크의 물리량을 측정하는 단위다. PDV는 사용자의 디지털 주권이다. OpenHash는 모든 소통의 불변하는 증인이다."**

## 2. 핵심 설계 원칙

| 원칙 | 설명 |
|------|------|
| **PDV 주권** | 모든 소통 데이터는 사용자 본인의 PDV에 저장. 중앙 서버·고팡 회사·L1 노드 누구도 내용에 접근 불가 |
| **OpenHash 불변성** | 모든 소통은 발생 즉시 해시가 OpenHash에 등록. 이후 내용·시각 변조 수학적 불가능 |
| **발신자 서명 책임** | 모든 메시지에 발신자 디지털 서명 첨부. 발송 사실 부인 불가 |
| **GUID 불변성** | 영구 식별자는 이사·기기 변경·닉네임 변경과 무관하게 유지 |
| **GDC 프로토콜 내재화** | GDC 상태가 주소·암호화·라우팅의 입력값으로 사용 |
| **위치 독립성** | 공개 IPv6는 가변, GUID와 분리된 별칭 계층으로 관리 |
| **L1 리던던시** | 읍면동당 최소 3개의 L1 노드 |
| **경제적 보안** | 모든 공격에 GDC 비용이 수반되도록 설계 |
| **점진적 프라이버시** | 기본 K-익명성 → Mixnet·PIR 선택 가능 |

## 3. 위협 모델

### 3.1 방어하는 공격자

| 공격자 유형 | 공격 방법 | GAS v1.6 대응 |
|------------|----------|--------------|
| 네트워크 도청자 | 패킷 스니핑, MitM | TLS 1.3 + E2EE |
| 악의적 L1 운영자 | 드롭·메타데이터 수집 | K=3 BFT + 슬래싱 |
| 국가급 트래픽 분석 | 전체 인터넷 상관분석 | Mixnet + PIR |
| Sybil 공격자 | 대량 계정으로 DHT 오염 | GDC 가중 거리 + 신뢰 모델 |
| 스팸 발신자 | 대량 저비용 메시지 | 오프라인 큐 예치금 + 적응형 PoW |
| **증거 인멸 시도** | **가해자가 자신의 PDV에서 메시지 삭제** | **피해자 PDV + 발신자 서명 + OpenHash 해시 체인** |
| **발송 부인** | **"내가 보낸 게 아니다" 주장** | **발신자 디지털 서명 — 수학적으로 부인 불가** |
| DDoS (스텔스 홍수) | 수신자 스캔 부하 급증 | GDC 소각 기반 태그 확장 |

### 3.2 방어하지 않는 공격자

| 공격 | 이유 |
|------|------|
| 기기 직접 물리 압수 | 기기 암호화로 일부 방어, 완전 방어 불가 |
| 양자 컴퓨터 | v2.0에서 PQ 암호 전환 예정 (§26) |
| 적법한 사법 집행 | 설계상 협조 채널 제공 (§20) |
| **수신자 없는 케이스의 분쟁** | **발생 빈도 극히 낮음. 비용 대비 효용 고려, 현행 법체계의 일반 원칙 적용** |

## 4. 3계층 ID 구조

```
L0 (글로벌)   : OpenHash 컨소시엄 원장 — GUID·닉네임·GDC 스테이킹 앵커링
L1 (지역)     : 읍면동 단위 노드 클러스터 (K=3 리던던시)
L2 (개인)     : 사용자 디바이스 (PDV, Personal Data Vault)
```

| 식별자 | 변경 가능성 | 용도 |
|--------|------------|------|
| **Nickname** (`@체제수리공`) | 가변 (30일 쿨다운, 경매 가능) | 인간 친화적 검색 |
| **Permanent GUID** | 불변 | 라우팅 앵커, GDC 계정 참조 |
| **GDC Master Public Key** | 교체 가능 (GUID 연결 유지) | GDC 잔액·스테이킹 소유 증명 |
| **Current IPv6** | 이사 시 변경 | 네트워크 도달성, 하위 니블에 신뢰 등급 인코딩 |
| **Trust Level** | 스테이킹 시 상향 | Sybil 저항, 기능 제한, IPv6 신뢰 비트 결정 |

## 5. IPv6 주소 체계

### 5.1 비트 레이아웃

```
비트  0~47  : OpenHash ULA prefix (fd6f:7068:6173::/48) — 고정
비트 48~51  : 버전 필드 (4비트, GAS v1.6 = 0x6)
비트 52~55  : fork_id (4비트, 메인넷 = 0x0)
비트 56~63  : 예약 (8비트)
비트 64~79  : 국가코드 (16비트)
비트 80~95  : 광역코드 (16비트)
비트 96~107 : 시군구코드 (12비트)
비트108~115 : 읍면동코드 (8비트)
비트116~123 : 순번 (8비트, 개인번호 상위)
비트124~127 : 신뢰 등급 인코딩 (4비트)
```

### 5.2 신뢰 등급 인코딩 (하위 니블)

```
trust_level_encoding (4비트):
  0x0 = L0 (익명, 스테이킹 0 GDC)
  0x1 = L1 (전화번호 인증, 스테이킹 0 GDC)
  0x2 = L2 (≥100 GDC 스테이킹)
  0x3 = L3 (≥1,000 GDC 스테이킹 + KYC)
  0x4~0xF = 미래 예약

개인번호 = (순번 << 4) | trust_level_encoding
```

**예시 주소:**

```
fd6f:7068:6173:6000:0082:0050:F013:A3F3
                │                     ↑
                └── 버전 0x6   trust_level 0x3 (L3)
```

## 6. Stealth Address

### 6.1 기본 구조

수신자 Bob: `(b, B)` (개인키, 공개키), `(v, V)` (뷰 키 쌍)

```
r      ← 난수
R      = r·G
P      = H(r·B)·G + B
tag    = H(r·V) mod 2^(32 + burn_amount_factor)
```

### 6.2 GDC 소각 기반 태그 확장

```
burn_amount_factor = floor( log₂( burned_GDC + 1 ) )   ← 최대 8

소각 0 GDC  → 태그 32비트 (기본, 충돌 확률 2^-32)
소각 7 GDC  → 태그 35비트
소각 255 GDC → 태그 40비트 (충돌 확률 2^-40)
```

**수신자 인센티브:** 수신자는 최소 소각량 요건을 설정하여 스팸·DDoS를 경제적으로 억제합니다.

### 6.3 수신자 스캔 최적화 (O(1))

```
expected_tag = H(R·v) mod 2^(32 + burn_amount_factor)
if tag == expected_tag → 내 메시지 → 개인키 b로 복호화
else                   → 스킵
```

## 7. 닉네임 등록, 경매 및 충돌 방지

### 7.1 Proof-of-Uniqueness 등록

```
[1] DHT 조회: SHA-256(nickname) → 결과 없어야 함
[2] 등록 수수료 소각: 최소 0.001 GDC
[3] L0 트랜잭션: { nickname_hash, permanent_guid, trust_level, user_sig }
[4] L0 노드 2/3 서명 확인 후 확정
```

### 7.2 닉네임 경매

```
경매 트리거: 닉네임 만료 30일 전, 타인의 입찰 발생 시

낙찰금의 80% → 이전 소유자 환불
낙찰금의 20% → 소각
경쟁 없을 경우: 갱신 비용 0.0001 GDC
```

### 7.3 닉네임 변경 및 동명이인 처리

```
변경 시:
  기존 닉네임 → 'transitioning' (30일) → 'expired' (90일 잠금)
  120일 후 → 경매 대상

동명이인: 시스템이 @체제수리공.제주, @체제수리공.2026 등 제안
```

## 8. 이동성 모델

## 13. GDC 가중 DHT 라우팅

### 13.1 경제적 거리 함수

```
거리(노드A, 노드B) = XOR( GUID_A, GUID_B ) - w × log₂( stake_GDC_B + 1 )
```

스테이킹이 높은 사용자 → DHT에서 더 가깝게 위치 → 검색 속도 향상.

## 20. 법적 컴플라이언스 아키텍처 (v1.6 전면 재정립)

### 20.1 설계 철학

> **"고팡에서 모든 소통은 발생하는 순간 증거가 된다. 별도의 증거 보존 절차가 필요 없다. PDV가 원본을 보관하고, 발신자 서명이 책임을 확정하며, OpenHash가 시각과 내용을 불변으로 기록한다."**

고팡의 증거 체계는 중앙 서버를 전제하는 기존 법률 IT 시스템의 논리와 근본적으로 다릅니다. 기존 시스템에서 증거를 보전하는 것은 "가해자가 서버 관리자에게 삭제를 요청할 수 있기 때문"이었습니다. 고팡에서는 그 전제가 성립하지 않습니다.

### 20.2 PDV+OpenHash 자기완결 증거 구조 (v1.6 핵심)

#### 20.2.1 세 가지 증거 요소

```
① 피해자 PDV 원본
   발신자(가해자)가 자신의 PDV에서 메시지를 삭제해도,
   수신자(피해자)의 PDV에 동일한 원본이 그대로 보관됩니다.
   소통은 쌍방향이며, 양측의 PDV에 동일한 메시지가 저장됩니다.

② 발신자 디지털 서명
   모든 메시지에는 발신자의 ECC 디지털 서명이 첨부됩니다.
   피해자 PDV의 원본에 이 서명이 포함되어 있으므로,
   가해자는 발송 사실을 수학적으로 부인할 수 없습니다.

③ OpenHash 해시 체인
   메시지 발송 즉시 내용의 SHA-256 해시가 OpenHash에 등록됩니다.
   h_i = SHA-256(h_{i-1} ∥ data_i ∥ height_i)
   등록 이후 내용·시각 변조는 수학적으로 불가능합니다.
   법원은 OpenHash 네트워크에서 독립적으로 검증할 수 있습니다.
```

#### 20.2.2 자기완결성의 의미

이 세 요소는 서로를 보완하여 어떤 시나리오에도 증거 구조가 유지됩니다.

| 시나리오 | 대응 |
|---------|------|
| 가해자가 자신의 PDV에서 삭제 | ① 피해자 PDV에 원본 존재 |
| "내가 보낸 게 아니다" 주장 | ② 발신자 디지털 서명으로 수학적 부인 불가 |
| "내용이 변조되었다" 주장 | ③ OpenHash 해시와 비교하여 즉시 검증 |
| "시각이 다르다" 주장 | ③ OpenHash 블록 높이로 발신 시각 확정 |
| 피해자가 내용을 위조하려 시도 | ② 발신자 서명 없이는 위조 불가 |

#### 20.2.3 Legal Hold가 불필요한 이유

v1.4~v1.5의 Legal Hold(PDV에서 메시지 삭제를 금지하는 스마트 컨트랙트)는 다음 가정에 기반했습니다.

> "가해자가 자신의 PDV에서 원본을 삭제하면 증거가 사라진다."

이 가정은 PDV+OpenHash 구조에서 성립하지 않습니다. 위 §20.2.2의 세 요소 조합으로 Legal Hold 없이도 동일한 증거 보전 효과가 달성됩니다. Legal Hold는 복잡성만 더하는 불필요한 레이어였습니다.

**수신자가 없거나 고팡 비사용자인 경우:** 이 케이스는 법적 분쟁으로 이어지는 빈도가 극히 낮습니다. 설령 분쟁이 발생하더라도 현행 전자서명법·전자문서법의 일반 원칙으로 처리하는 것이 비용 대비 효용 측면에서 합리적입니다. 이를 위한 별도 시스템을 구성할 필요가 없습니다.

### 20.3 ZKP 소유권 인증: 완전 수식화 (Schnorr Protocol)

#### 파라미터

| 기호 | 정의 |
|------|------|
| `G` | 타원곡선 생성자 (Curve25519) |
| `n` | 타원곡선 군의 위수 |
| `sk` | 사용자 개인키 (비공개) |
| `pk = sk·G` | 사용자 공개키 |
| `H_m` | 확인 대상 메시지의 SHA-256 해시 |
| `k` | 증명자의 임시 난수 |
| `R = k·G` | Commitment |
| `c` | Challenge (법원 제시 또는 Fiat-Shamir) |
| `s` | Response |
| `π = (R, s)` | 최종 ZKP 증명 |

#### 프로토콜

```
─────────────────────────────────────────────
  사용자 (Prover)          법원 (Verifier)
─────────────────────────────────────────────
[Step 1: Commit]
  k ← rand(1, n-1)
  R = k·G
  사용자 → 법원: (R, H_m, pk)

[Step 2: Challenge]
                         c ← rand(1, n-1)
                         법원 → 사용자: c

[Step 3: Response]
  s = (k + c·sk) mod n
  사용자 → 법원: s

[Step 4: Verification]
                         s·G =?= R + c·pk
                         ∵ s·G = k·G + c·(sk·G)
                               = R   + c·pk  ✓
─────────────────────────────────────────────
```

**비대화형 변환 (Fiat-Shamir):**

```
c = H(pk || R || H_m || Nonce || timestamp)
s = (k + c·sk) mod n
π = (R, s, c, H_m, Nonce, timestamp)
```

### 20.4 ZKP 유료 증명 및 증명 무게

#### 유료 증명

```
Verification API 사용 수수료: 0.01 GDC / 1회 검증
  - 무분별한 개인정보 요청 방지
  - 네트워크 유지 비용 충당
  - 수수료는 검증에 참여한 L1 노드 운영자에게 배분

사용자 자신의 ZKP 생성: 무료
```

#### 증명의 무게 (Proof Weight)

```
ZKP 증명 π에 발신자가 소각한 GDC 양을 선택적으로 포함:

proof_weight = burned_GDC_for_proof

proof_weight = 0       → 표준 검증 절차
proof_weight = 10 GDC  → 추가 조사 없이 소유권 인정 권고
proof_weight = 100 GDC → 높은 신뢰도, 즉각 인정 권고
```

### 20.5 OpenHash 타임스탬프 앵커링

모든 메시지는 발송 즉시 OpenHash에 해시가 등록됩니다. 이것이 고팡의 기본 동작이며, 별도 활성화가 필요하지 않습니다.

```
모든 메시지 (기본):
  발송 즉시 → SHA-256(메시지_내용) → OpenHash 해시 체인에 등록
  h_i = SHA-256(h_{i-1} ∥ content_hash ∥ block_height)

배치 앵커링 (1시간 주기):
  수집된 해시들 → Merkle Tree → Root를 Openhash 메인넷 블록에 기록

법원 독립 검증:
  메시지 해시 + Merkle Proof → 온체인 Root와 비교
  → 생성 시각과 내용의 불변성 수학적 확인
```

**Time-drift 대응:**

```
T_device : 발신자 PDV 로컬 시계 (신뢰도 低, 조작 가능)
T_l1     : L1 노드 3개 수신 타임스탬프 중앙값 (NTP ±500ms)
T_anchor : OpenHash 블록 타임스탬프 (신뢰도 最高, 변경 불가)

신뢰 우선순위: T_anchor > T_l1 > T_device

검증:
  |T_anchor - T_l1| > 30초 → Suspicious
  T_device > T_l1 + 5초   → Flagged
  모두 통과               → Valid
```

### 20.6 선택적 법적 에스크로 (Trust Level L3)

Trust Level L3 사용자가 자발적으로 활성화하는 옵션입니다.

```
백업 복호화 키 분할: Shamir Secret Sharing (3-of-3)
  share_1 → 국내 법무법인 A
  share_2 → 해외 중립 기관 B
  share_3 → 사용자 PDV 백업

복호화 조건: 법원 영장 + 2개 이상의 share 결합
단일 기관의 단독 복호화 불가
```

에스크로 비활성 상태(기본)에서의 법적 리스크는 사용자 본인이 부담함을 앱에서 명시합니다.

### 20.7 L1 노드 Safe Harbor

| 요건 | 내용 |
|------|------|
| 무지성 | E2EE로 메시지 내용 열람 불가 |
| 자동화 | 수동 개입 없는 자동화 시스템 |
| 적법 요청 협조 | 영장에 따라 라우팅 메타데이터 제공 가능 (§20.9 API) |
| 신고 창구 | 불법 콘텐츠 신고 접수 및 검토 절차 |
| 로그 최소화 | 내용 로그 없음, 라우팅 메타데이터만 최소 보관 |

**L1 노드가 보관하는 것:** CID(메시지 저장 위치), 라우팅 타임스탬프, 발신·수신 주소
**L1 노드가 보관하지 않는 것:** 메시지 내용, 사용자 실제 신원 정보

### 20.8 사회적 복구의 법적 효력 명세

| 관점 | 분석 |
|------|------|
| 대리권 유사성 | 사전 동의 기반 → 임의대리(민법 §114) 유사 |
| 상속권 유사성 | 사망 시 사전 지정 → 유언 대용 신탁 유사 |

**공증 옵션:**
- 옵션 A: 네트워크 내 기술적 복구만 (법적 효력 미보장)
- 옵션 B: 공증인 앞에서 '복구 위임장' 작성 후 GAS에 해시 등록
- 옵션 C: 유언장에 GAS 복구 절차 명시

사회적 복구가 보장하지 않는 것: 이전 개인키로 암호화된 메시지 복호화, 법적 소유권 이전, GDC 토큰 재산적 가치 이전.

### 20.9 법원 전용 Verification API

```
기본 URL: https://court-api.gopang.net/v1/
인증: 법원 공개키로 서명된 JWT
TLS: 1.3, 양방향 인증서 검증
수수료: 0.01 GDC / 1회 검증 (자동 차감)
```

**엔드포인트 (v1.6 개정):**

```http
POST /verify/ownership          → ZKP 소유권 검증
POST /verify/signature          → 발신자 디지털 서명 검증
GET  /evidence-report/{id}      → 증거 일관성 보고서
GET  /proof-weight/{id}         → 증명 무게 조회
GET  /openhash/verify/{hash}    → OpenHash 해시 독립 검증
```

*Legal Hold 관련 엔드포인트(`/legal-hold/apply`, `/legal-hold/release`) 제거.*

**증거 일관성 보고서 포함 항목 (v1.6 개정):**

```
1. 메시지 식별 정보   : 고유번호, 저장 위치(CID), 발신자 주소
2. 타임스탬프        : T_device, T_l1(중앙값), T_anchor, 상호 편차
3. 시각 신뢰도 판정  : Valid / Warning / Suspicious / Flagged
4. OpenHash 검증    : 해시 체인 재계산 결과 (변조 여부)
5. 발신자 서명 검증  : 디지털 서명 유효성
6. ZKP 소유권 증명  : π = (R, s, c, H_m) (선택적)
7. 증명 무게        : 발신자 소각 GDC 금액 (선택적)
8. 종합 일관성 점수  : 0~1 (참고값)
```

**접근 감사:** 모든 API 호출은 OpenHash 메인넷에 불변 기록됩니다.

### 20.10 디지털 증거 실증 시나리오 (v1.6 개정)

업무상 배임 분쟁을 예시로, Legal Hold 없이 완결되는 증거 제출 흐름입니다.

```
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
  당사자              GAS 네트워크          법원
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
[D+0]  원고가 피고에게 메시지 전송
       발송 즉시:
         ① 피고(수신자) PDV에 원본 + 원고 디지털 서명 저장
         ② OpenHash: SHA-256(메시지) 해시 체인 등록 (T_anchor)
         ③ 원고 PDV에도 동일본 저장

[D+30] 분쟁 발생. 피고가 자신의 PDV에서 메시지 삭제 시도
         → 자신의 PDV에서는 삭제 가능
         → 그러나 원고(피해자) PDV의 원본은 영향 없음
         → OpenHash 해시 체인은 이미 불변 등록 완료

[D+30] 원고 변호인이 원고의 PDV에서 메시지 추출
         → 증거 패키지 생성:
            메시지 원본 + 피고 디지털 서명 + OpenHash 해시

[D+45] 법원 Verification API 호출 (0.01 GDC/회)
         POST /verify/signature → 피고 서명 유효 확인
         GET /openhash/verify/{hash} → 해시 체인 독립 검증
         → T_anchor 기준 발송 시각 확정
         → 내용 변조 없음 확인

[D+60] 법정 제출
         원본 메시지 + 피고 서명 + OpenHash 검증 결과 + 증거 보고서
         → 피고는 서명이 첨부된 메시지에 대해 발송 부인 불가

[D+90] 판결
━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━━
```

**기술적 보장이 사법적 정의로 이어지는 경로:**

```
[기술 계층]                  [법적 효력]

피해자 PDV 원본    →  증거 원본 보전 (가해자 삭제와 무관)
발신자 디지털 서명 →  발송 사실 확정 (부인 불가)
OpenHash 해시 체인 →  내용 무결성·발송 시각 (불변 기록)
ZKP 소유권 증명   →  계정 소유권 확인 (선택적)
Verification API  →  절차적 정당성 (법원 직접 검증)
증거 일관성 보고서 →  맥락 무결성
```
